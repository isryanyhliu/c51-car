### ② 资料-引脚定义
手册原文：  
![](../../assets/images/Pasted%20image%2020260426171055.png)

**翻译 + 大白话：**
- **ENA = 左电机使能（对应你代码：Left_moto_pwm）**
- **ENB = 右电机使能（对应你代码：Right_moto_pwm）**
- **ENA = 低电平（0）→ 左电机直接禁用、停转**
- **ENA = 高电平（1）→ 左电机允许工作**
- ENB 同理

L298 驱动电机分 **两个完全独立的控制**：

|引脚|名字|作用|控制内容|
|---|---|---|---|
|IN1 / IN2|方向引脚|控制电机**正转 / 反转 / 停止**|方向|
|ENA|使能 / 调速引脚|控制电机**开 / 关 + 速度**|转速|

**PWM 调速**
- 给 ENA 一直输 **高电平 1** → 电机**全速转**
- 给 ENA 一直输 **低电平 0** → 电机**停止**
- 给 ENA **快速切换高低电平（PWM）** → 电机**调速**（高电平时间越长，速度越快）

**IN1/IN2 管方向，ENA 管速度和开关****
**写驱动代码 = 看芯片手册 + 看板子丝印，自己定义头文件**


### ③ 引脚+丝印：头文件定义

```c
// 完全来自：L298手册 + 小车硬件丝印 
sbit Left_moto_pwm = P1^5; // ENA 左电机使能+调速 
sbit IN1 = P1^4; // 左方向 
sbit IN2 = P1^3; // 左方向 
sbit Right_moto_pwm = P1^0; // ENB 右电机使能+调速 
sbit IN3 = P1^2; // 右方向 
sbit IN4 = P1^1; // 右方向
```

  
>[!IMPORTANT]  
>所以，头文件相当于说明书  
  
  
>[!CAUTION]  
>为什么硬件生产商不直接把头文件给我们，而是给一个操作手册而已


**芯片厂商只生产「通用硬件」，不生产「你的板子」；

接线是开发板厂家自定义的，没有统一标准，所以不可能给你现成头文件。

1.芯片原厂（ST、Atmel、乐鑫等，比如 L298、51 内核）只负责出**数据手册 (Datasheet)**：
- 芯片电气规则
- 引脚功能定义（ENA 是使能、IN1 是方向）
- 真值表、电平逻辑、时序
**绝对不能写死引脚**

比如 L298 的 ENA：
- 你的小车：ENA → P1^5
- 别家开发板：ENA → P2^0
- 竞赛板子：ENA → P3^1

如果原厂写死头文件 `sbit ENA = P1^5;`换个板子直接报废，芯片就没法通用售卖。


**开发板 / 小车小厂（做你这块 QX_A11 板子的厂家）**

他们**明明可以给完整头文件 + 全套代码**，但为什么不给？

1. **硬件迭代极快**
    改一版板子、换个走线、换引脚，旧头文件直接作废。  
    手册不用改，头文件要全改，维护麻烦。  
2. **故意留白，留给二次开发**
    全都给你写死了，还怎么上课、怎么让学生练引脚配置？  
    

**头文件本来就该开发者自己写**，这是嵌入式全世界统一的规矩：
- 芯片厂：给**功能规则**（手册）【**引脚功能**】
- 板子厂：给**丝印接线**（板上印 P16、P15、ENA）【**引脚位置**】
- **开发者：自己结合两者，手写引脚头文件**

### ④ 软件PWM vs 定时器PWM

✅ **软件 PWM**：**CPU 亲自盯着**，不停开关引脚，**啥也不能干，专门生成 PWM**  
✅ **定时器 PWM**：**硬件定时器自动干活**，CPU 该干嘛干嘛，**完全不耽误主程序**  


**工作原理**

1. 软件 PWM（你现在用的）  
	- 靠 `Delay_us` 延时函数**手动控制**ENA/ENB 引脚高低电平  
	- **CPU 全程阻塞**，执行 PWM 的时候，不能执行别的代码   
	- 代码简单，纯靠循环延时  

2. 定时器 PWM（工业 / 比赛标准用法）  
	- 用 51 内部**硬件定时器**自动计时，时间到了自动翻转引脚  
	- **CPU 完全解放**，主程序可以同时跑按键、LED、传感器  
	- 精度高、不受主程序延时影响  

| 对比项        | 软件 PWM               | 定时器 PWM        |
| ---------- | -------------------- | -------------- |
| **谁来执行**   | CPU 软件延时             | 硬件定时器 自动执行     |
| **占用 CPU** | 占满 CPU，不能干别的事        | 0 占用，CPU 自由运行  |
| **精度**     | 低，受主程序延时干扰           | 极高，稳定无误差       |
| **代码难度**   | 极低，新手秒懂              | 中等，要懂定时器中断     |
| **多任务**    | 不能同时控制电机 + 读按键 / 传感器 | 可以同时跑所有功能      |
| **小车表现**   | 调速时会卡顿、偶尔抽搐          | 平稳顺滑，无卡顿       |
| **适用场景**   | 教学入门、演示原理            | 正式运行、循迹 / 避障小车 |

## 2.1 拓展：定时器PWM调速

### ① 寄存器

- **TMOD**：全称 Timer Mode（定时器模式寄存器）。  
  它是用来**设置定时器工作方式**的控制面板。  
- **TH0 和 TL0**：全称 Timer High 0 和 Timer Low 0。  
  它们两个合在一起，是定时器0的**计数水桶**。  

### ② TMOD 为什么等于 0x01

前面说了，单片机是一台机器。我们要让“定时器0”工作在“16位模式”（这个模式最常用）。  
我们该怎么告诉单片机呢？  

这时候就要翻开《数据手册》查 TMOD 寄存器了。  
手册上会画一个表格，告诉你 TMOD 这个寄存器有 8 个开关（8个位）：  
![](../../assets/images/Pasted%20image%2020260427212610.png)

我们要用的是**定时器0**，所以只看右边的 4 个开关（位 3 到位 0）。  
手册上写着：如果要用 16位定时器模式，需要把 M1 置为 0，M0 置为 1。其他不用的开关全关掉（置为 0）。  
  
所以，这 8 个开关的状态就变成了： **`0 0 0 0 0 0 0 1`**（这是一个二进制数）  
  
在 C 语言里，写一长串 0 和 1 太麻烦了。  
程序员习惯把二进制转换成**十六进制**（以 `0x` 开头）  
`0000 0001` 转换成十六进制，就是 **`0x01`**  

这就是 `TMOD |= 0x01;` 的由来！  
  
翻译成人话就是：**照着说明书，  
把 TMOD 寄存器的最右边那个开关打开，让定时器0准备干活**

### ③ TH0 和 TL0 是什么？0xFF 和 0x9C 又是哪来的？

定时器说白了就是一个**数数机器**，你可以把它想象成一个会自动加水的**水桶**。  
在16位模式下，这个水桶的容量是 **65536** 滴水。  
单片机每隔极短的时间（对于11.0592M晶振，大约是 1微秒）往里滴一滴水。  
当滴满 65536 滴水时，水漫出来了（这叫**溢出**），单片机就会报警（触发**中断**）  
  
**现在问题来了：**  
如果你让一个空桶（从 0 开始）滴水，滴满 65536 滴大概需要 71 毫秒  
这对控制电机来说**太慢了**！  
我们希望它滴 **100滴水** 就立刻报警一次（大概只需要 100微秒），怎么办？
  
**聪明的做法是：提前往桶里倒水！**  
容量是 65536，我希望再加 100 滴就满  
那我一开始应该在桶里放多少水？ 65536 - 100 = **65436**。  
  
这就是我们要设置的**初始值**。  
只要我把 65436 塞进定时器里，它再数 100 个数就会报警。  

但单片机不认十进制的 65436，我们要把它转成十六进制。  
打开电脑自带的计算器，选择“程序员”模式，输入 65436，  
你会看到它的十六进制是：**`FF9C`**。  
  
因为单片机是 8 位的，一个水桶装不下这么大的数，所以这个大水桶被拆成了两个小水桶：
- **TH0**（高位水桶）：装前两位，所以 `TH0 = 0xFF;`  
- **TL0**（低位水桶）：装后两位，所以 `TL0 = 0x9C;`  

当你写下：  
```c
TH0  = 0xFF;  
TL0  = 0x9C;  
```
  
你其实是在告诉单片机：  
**“兄弟，定时器水桶已经预先给你装了 65436 滴水了，你再往里滴 100 滴就赶紧叫我！”**

>[!TIP]
>表格的第一列就是我们在代码里直接写的名字。  
>为什么在 C 语言里直接写 `TMOD` 单片机就能懂？  
>因为在程序的开头通常有一句 `#include <reg52.h>`  
>这个头文件就像一本字典，它告诉编译器：  
>当程序员写 `TMOD` 时，指的就是地址为 `89H` 的那个储物格。

### ⑤ 总结代码编写方法
- **想做什么？**（比如：我要用定时器 0）  
- **查手册：** 找到名字叫 `TMOD` 的控制台，看看怎么配置它  
- **查地址：** 确认它的地址和内部位的功能  
- **算数值：** 计算出要填进去的十六进制数（比如 `0xFF9C`）  
- **写程序：** `TMOD = 0x01; TH0 = 0xFF; TL0 = 0x9C;`  



## 2.2 PWM调速 Gemini 解释

没问题！咱们彻底抛开那些深奥的术语，把这套代码想象成  
**“在厨房里通过控制开关火，来精准熬粥”** 的过程  
  
要把这辆 C51 小车跑起来，代码其实只做了 **三件事**：  
  
### ① 第一步：买个精准的“闹钟”  
**对应代码：** `TMOD = 0x01;`  
  
单片机就像一个大工厂，里面有很多设备。`TMOD` 就是**设备管理清单**。  
  
- **动作：** 你对着手册（地图）查到，把这个清单的最后一位改成 `1`（即 `0x01`），就相当于给厨房下单买了一个 **“16位高精度闹钟”**。  
- **为什么要买它？** 因为我们要用这个闹钟来卡时间，每隔一丁点时间就切换一次电机的开关。  


### ② 第二步：给闹钟“调时间”

**对应代码：** `TH0 = 0xFF; TL0 = 0x9C;`

这个闹钟有个怪脾气：它只能从某个数字往上数，数到 65536 才会响。

- **现状：** 如果从 0 开始数，数到响要很久，小车反应太慢。  
- **绝招：** 我们手动给它加点“预付款”。就像倒计时，我们要它数 100 下就响，那我们就先给它填到 65436（十六进制就是 `FF9C`）。  
- **结果：** 闹钟一启动，数 100 下，**叮！** 响了。这大约只用了 100 微秒（也就是万分之一秒）。  

### ③ 第三步：闹钟响了怎么办？（疯狂的“开关操作员”）

**对应代码：** `void Timer0_ISR(void) interrupt 1`

这是整段代码最天才的地方。我们聘请了一个 **“极其听话的操作员”**，专门等闹钟响。

1. **闹钟响：** 操作员立刻冲进厨房。  
2. **重置闹钟：** 赶紧把 `FF9C` 重新填进去（保证下一次还是 100 下就响）。  
3. **报数：** 操作员手里有个计数器 `pwm_cnt`，每次响就 **+1**。这个数在 0 到 99 之间循环。  
4. **做决定（核心）：**  
    - 假设你想要 30% 的速度。  
    - 操作员看一眼计数器：如果现在报数是 **0~29**，就把电机的**火打开**（高电平）。  
    - 如果报数是 **30~99**，就把电机的**火关掉**（低电平）。  

### ④ 最后的“魔法”效果

因为操作员动作极快（每秒钟进厨房上万次），电机根本感觉不到火在一开一关。  
- **电机视角：** “虽然一会儿有电一会儿没电，但平均下来，我感觉力气只有平时的 30%。”  
- **结果：** 小车稳稳地以 30% 的速度跑了起来！  

### ⑤ 总结逻辑

1. **翻手册：** 找到 `TMOD` 这个开关地址，查到 `0x01` 是开启闹钟模式。  
2. **算算术：** 算出你想让闹钟多久响一次，把数字拆成高位（`TH0`）和低位（`TL0`）。  
3. **写逻辑：** 在 `interrupt 1`（闹钟专用办公室）里，写下“如果计数器没到预设值就开灯，到了就关灯”的逻辑。  

**一句话总结：** 这个代码就是利用一个**疯狂报数的中断闹钟**，通过**高频率地切换电源开关**，用“骗”的方式实现了对电机速度的控制。
